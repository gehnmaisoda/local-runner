# 他ツールとの比較

LocalRunner は「macOS 上で開発者が自分のスクリプトを定期実行したい」というユースケースに特化している。既存の選択肢と比較して、どこに位置するかを整理する。

## 比較表

|  | cron | launchd | n8n / Airflow | **LocalRunner** |
|:---|:---:|:---:|:---:|:---:|
| スケジュール | cron 式 | plist で定義 | cron 式 + DAG / ワークフロー | cron 式 + プリセット |
| 実行ログの永続化 | -- | stdout/stderr のみ | Web UI で閲覧可 | JSON + CLI / Web UI |
| 失敗通知 | -- | -- | Slack / Email 等 | Slack (Bot Token / Webhook) |
| Web UI | -- | -- | あり | あり |
| ローカル認証情報 | そのまま使える | そのまま使える | 再設定が必要 | そのまま使える |
| macOS スリープ復帰 | 実行されない | 制限あり | 実行されない | タスク単位で設定可 |
| ネットワーク復帰待ち | -- | -- | -- | 自動 |
| セットアップ | OS 組み込み | OS 組み込み (XML 手書き) | Docker or Node.js + DB | `brew install` |
| 料金 | 無料 | 無料 | セルフホスト無料 / クラウド有料 | 無料 |

---

## cron

macOS / Linux に標準搭載されるタスクスケジューラ。`crontab -e` でスケジュールを定義し、指定時刻にコマンドを実行する。

**強み**:
- OS 組み込みで追加インストール不要
- cron 式による柔軟なスケジュール定義
- 枯れた技術で安定性が高い

**限界**:
- **ログ管理がない** — stdout/stderr は自前でリダイレクトする必要がある。実行履歴の閲覧・検索手段はない
- **失敗通知がない** — コマンドが失敗しても気づけない。通知が欲しければスクリプト内で自前実装する
- **スリープ中のスケジュールはスキップされる** — macOS でノート PC を閉じていた間のジョブは復帰後も実行されない。キャッチアップの仕組みはない
- **macOS では非推奨** — Apple は launchd への移行を推奨しており、将来的なサポートは不透明

## launchd

macOS 標準のサービス管理フレームワーク。Apple が cron の後継として推奨している。XML 形式の plist ファイルでジョブを定義し、`launchctl` で管理する。

**強み**:
- macOS ネイティブで OS との統合度が高い
- `StartCalendarInterval` によるスリープ復帰後の実行に部分対応
- `KeepAlive` や `WatchPaths` など多彩なトリガー条件

**限界**:
- **plist (XML) の手書きが煩雑** — 単純な「毎時実行」でも 20 行以上の XML を書く必要がある
- **実行履歴の管理機能がない** — ログは stdout/stderr のファイルリダイレクトのみ。「いつ実行されて成功したか」の履歴は残らない
- **失敗通知がない** — ジョブが失敗しても通知する仕組みがなく、ログを目視確認するしかない
- **管理 UI がない** — `launchctl` コマンドのみ。一覧の見やすさや操作性は最低限

## n8n / Airflow

ワークフローオーケストレーションツール。タスクの依存関係 (DAG) や条件分岐を含む複雑なパイプラインを構築・管理できる。

- **n8n**: ノーコード / ローコードのワークフロー自動化プラットフォーム。セルフホスト Community Edition は無料 (Sustainable Use License)。クラウド版は Starter で月額 20 ドル〜
- **Apache Airflow**: Python ベースの OSS ワークフローエンジン (Apache License 2.0)。データパイプラインや ETL 処理で広く使われている

**強み**:
- **DAG によるワークフロー管理** — 「タスク A → B → C」のような依存関係、条件分岐、リトライポリシーを定義できる
- **豊富なインテグレーション** — n8n は 400+、Airflow は多数のオペレーターで外部サービスと連携
- **Web UI が充実** — 実行履歴、ログ、ワークフローの視覚的な管理

**限界**:
- **セットアップが重い** — n8n は Docker または Node.js + DB (デフォルト SQLite、本番は PostgreSQL 推奨)。Airflow は Web サーバー + スケジューラー + ワーカー + メタデータ DB + メッセージキューが必要
- **ローカルの認証情報が使えない** — Docker コンテナ内で動作するため、ホストマシンの `gh`, `gcloud`, `aws` CLI 等の認証情報はそのまま使えない。ボリュームマウントで渡すか、ツール内のクレデンシャル管理に登録し直す必要がある。**「ターミナルで動くコマンドがそのまま動く」とはいかない**
- **macOS のスリープに対応していない** — n8n はスリープ中のスケジュールをスキップし、キャッチアップ機能は未実装。Airflow は `catchup=True` があるがサーバープロセス稼働中が前提で、macOS スリープでプロセスごと停止する場合は機能しない
- **個人のローカルタスクにはオーバースペック** — 「毎朝スクリプトを 1 本回したい」に対して DAG やワーカー管理は不要。設計思想がチーム向け・サーバー向けであり、ノート PC で開け閉めしながら使う想定ではない

## LocalRunner の位置づけ

LocalRunner は上記の空白地帯を埋める。

**cron / launchd に対して**: ログ・通知・Web UI を追加し、「実行するだけ」から「管理できる」へ引き上げる。

**n8n / Airflow に対して**: Docker / DB / 認証再設定のオーバーヘッドを排除し、`brew install` 一発で同等の基本機能を提供する。

**LocalRunner だけの特徴**:
- **ローカル認証情報がそのまま使える** — `zsh -l -c` でユーザーのログインシェルとして実行するため、`gh`, `gcloud`, `aws`, `git` 等の CLI 認証がそのまま通る。追加の認証設定はゼロ
- **macOS ネイティブ統合** — LaunchAgent で OS 起動時に自動起動。スリープ復帰時のキャッチアップ実行、ネットワーク復帰待ちをネイティブサポート
- **ゼロインフラ** — Docker 不要、DB 不要。`brew install` → `lr install` の 2 コマンドで完了
- **無料** — MIT ライセンスの OSS。利用制限なし
